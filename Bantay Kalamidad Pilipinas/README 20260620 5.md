# Rescue Dashboard — Scrollable Multi-Item Tiles

All three tiles on the rescue dashboard now show every matching row, with a
visible scrollbar when content overflows the tile's height.

## Files in this folder

| File | What changed |
|---|---|
| `rescuedashboard_mainlayout_view.xaml` | All 3 tiles now use `ItemsControl` + `ScrollViewer` (Auto scrollbar) |
| `rescue_dashboard_mainlayout_ViewModel.cs` | 2 of the 3 backing properties rebuilt as collections |
| `AssignedOperation.cs` | **New model** — one row per assigned operation |
| `CurrentRescueOperationSummary.cs` | **New model** — one row per current operation |

## What changed and why

**Active Disaster Event** was already a collection (`DisasterEvent`) from
the last pass — it only needed the scrollbar swapped from `Hidden` to
`Auto`. No backend change.

**My Assigned Operations** and **Current Rescue Operations** were both
built around `SELECT TOP 1` last time, showing only the volunteer's most
recent operation. Making these scrollable meant they needed to actually be
lists first — a scrollbar on a single row doesn't do anything. Both are now:

- `AssignedOperations` — every operation this volunteer is assigned to
  (role, location, ID, status), no row limit, newest first.
- `CurrentRescueOperations` — every operation this volunteer is part of,
  one row per operation (ID, status, location, date started), newest first.
  This replaces the old "2 Ongoing, 1 Completed" aggregated breakdown
  string — now that you can see more than one row, an actual list is more
  useful than a summary.

Both query methods were renamed to reflect that they now return lists:
`InitializeAssignedOperationsCardList()` and
`InitializeCurrentRescueOperationsCardList()`.

## Visual consistency

All three tiles now use the exact same structure — `ScrollViewer` with
`VerticalScrollBarVisibility="Auto"` wrapping an `ItemsControl` with a
`DataTemplate`. Each row shows a status badge in the same top-right spot the
Active Disaster Event tile already used, so the three tiles look and behave
consistently.

## No SQL changes needed

Both new queries use `GetTableDataWithCustomizedQuery` with the same join
logic as before — just without the `TOP 1` and `ORDER BY` limiting it to one
row. No new stored functions or schema changes required.

## To apply

Replace the 2 `.cs` files and the `.xaml` file in your project with the
versions here. No SQL script needs to be re-run for this particular change.
