## The big one: rescue dashboard showed almost nothing

Same root cause as the donation dashboard bug from last time. I checked
`rescuedashboard_mainlayout_view.xaml` against the ViewModel directly and
found it binds to a completely different, larger set of properties:

| Card | XAML binds to | Existed before? |
|---|---|---|
| My Assigned Operations | `AssignedOperationRole`, `AssignedOperationLocation`, `AssignedOperationId`, `AssignedOperationStatus` | **No** |
| Active Disaster Event | `DisasterEvent` (ItemsControl source), with per-item `ActiveDisasterEvent`, `ActiveDisasterEventStatus`, `ActiveDisasterEventId`, `ActiveDisasterStartDate` | **No** — only a single flat string existed, wrong shape entirely |
| Current Rescue Operations | `CurrentRescueStatusBreakdown`, `LatestDispatch`, `CurrentOperationLocation`, `CurrentOperationDateStarted` | **No** |

`AssignedOperationsCount` and `CurrentOperationsCount` — the two counters
that were actually being computed correctly — aren't referenced **anywhere**
in this XAML at all. That's why the rescue dashboard showed literally
nothing on any of its three cards, worse than the donation side where at
least one number got through.

### What I added
- All 7 missing string properties, each backed by a real query.
- A new `RescueActiveDisasterEvent` model, since the "Active Disaster Event"
  card is an `ItemsControl` (a list), not a single value — the old
  `ActiveDisasterEvent` string property was structurally the wrong shape to
  ever work with that XAML, independent of naming.
- Extended `GetActiveDisasterEvent()` to return `Event_ID`, `Start_Date`,
  and a computed `Status` ('Ongoing' if any operation against that event is
  still active, 'Monitoring' otherwise) — the Day 1 version only returned
  the event name.
- "My Assigned Operations" shows the volunteer's most recently started
  assignment (role, location, ID, status) — the card has room for one
  operation's details, not a list, so this picks the latest by start date.
- "Current Rescue Operations" shows a status breakdown across all the
  volunteer's operations (e.g. "2 Ongoing, 1 Completed") plus their most
  recent dispatch location/date.

## MyTeam: a real missing column, now wired up

The XAML has a 4th grid column using a `PriorityBinding` trying `Contact`,
`ContactNumber`, `Contact_Number`, `VolunteerContact` in sequence — none of
which were ever supplied. I checked the schema: `Volunteer.Contact_Number`
is a real column. Added it to the `Member` model and to
`GetMyTeamByOperationID`'s return columns, so that column now populates for
real.

## Announcements: built from scratch, with a new table

There was no ViewModel and no `DataContext` wiring for this screen at all —
confirmed in the code-behind. The XAML turned out to be simpler than I
assumed back on Day 1: it's two single text fields (Urgent / Update), not a
list, so my original Day 1 plan (an `ObservableCollection<Announcement>`)
would have been the wrong shape.

There's no table anywhere in your schema that could supply this content, so
per your instruction I added one — `Announcements_Schema.sql` creates the
table plus a `GetLatestAnnouncements()` function that returns the most
recent Urgent and Update message in a single row. No admin UI for *writing*
announcements is included (out of scope); insert rows manually via SSMS for
now — an example `INSERT` is in the script's comments.

## False "error" popups removed

Same pattern as the donation side: `MyRescueOperations`, `MyTeam`, and
`RescueLocations` all showed a MessageBox like "No available rescue
operations" on every zero-result query — including the normal case of a
volunteer who hasn't been assigned anything yet. Removed; empty lists now
just render as empty.

## Verified correct, no changes needed

I checked these against their XAML and found they actually already
matched — worth noting since I was wrong about a couple of these on first
glance and corrected myself before changing anything:
- `RescueOperation` model (`OperationId`, `Event`, `Location`, `DateStarted`,
  `Status`) — matches `MyRescueOperations` XAML exactly.
- `RescueLocation` model (`Barangay`, `City`, `Province`, `EvacuationArea`,
  `RescueStatus`) — matches `RescueLocations` XAML exactly, and the
  ViewModel's constructor argument order is correct.
- `Member` model's existing 3 properties (`Volunteer`, `Role`, `Status`) —
  match `MyTeam` XAML exactly.
- The `@username` vs `@Username` casing difference in
  `GetRescueOperationsByUsername` — confirmed via Microsoft's own docs that
  `SqlParameter` names are not case-sensitive, so this was never a bug.

## Flagged, not fixed — needs your decision

**`ForgotPasswordCommand` and `GoogleLoginCommand`** are referenced in both
the donation and rescue login/signup XAML files, but neither command exists
in any ViewModel. Unlike the other gaps in this pass, these aren't "the
ViewModel forgot to expose a property the UI needed" — they're two
genuinely separate, unbuilt features (a password-reset flow and Google
OAuth integration). Building either is a real scope decision, not a bug fix,
so I left them alone rather than guess at what you want there. Buttons bound
to a missing `ICommand` just render disabled/inert — no crash, no visible
error, which is likely why this hasn't come up yet.
