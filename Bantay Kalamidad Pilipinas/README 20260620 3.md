# Real root cause found: XAML/ViewModel property name mismatch

Your donor1 diagnostic results were the key. Steps 1–5 all came back correct
— `dbo.TotalDonationsCount('donor1')` called directly in SSMS returns 2,
exactly as it should. That proved the SQL function, the join chain, and the
data itself were never the problem. The bug was somewhere else entirely.

## What was actually wrong

I checked `donationdashboard_mainlayout_view.xaml` directly and compared it
against the ViewModel. The XAML's dashboard cards bind to a much larger set
of properties than the ViewModel ever defined:

| Card | XAML binds to | Existed in ViewModel? |
|---|---|---|
| My Total Donations | `TotalItemCount`, `LastDonationDate`, `ImpactSummary` | **No** |
| Active Pledges | `ActivePledgesCount`, `NextPendingItem`, `PledgeCompletionRate` | Only `ActivePledgesCount` |
| Scheduled Pickups | `NextPickupDateTime`, `PickupLocation` | **No** |
| Scheduled Deliveries | `NextDropOffDateTime`, `DestinationHub`, `DeliveryStatus` | **No** |

`TotalDonationsCount` and `ScheduledPickupsCount` were being computed
correctly in the ViewModel this whole time — they just weren't bound to
anything in the XAML, so nothing ever displayed them. WPF bindings to a
missing property fail completely silently (no exception, no console
warning visible at runtime in Release builds — Debug output shows a binding
error, but it's easy to miss). `ActivePledgesCount` was the *only* number
that ever had a chance of appearing, because it's the only property name
that happened to match between the XAML and the ViewModel.

That's why donor1's 2 real donations never showed: there was no UI element
bound to a property that reflects them. `TotalDonationsCount` was correctly
sitting at `2` in memory; the screen just never asked for it under that name.

## What I did

Added all seven missing properties to `donationdashboard_mainlayout_ViewModel.cs`,
each backed by a real query:

- **`TotalItemCount`** — sum of `Quantity_Received` across all the donor's
  `Donated Items`, joined through `Donation`.
- **`LastDonationDate`** — most recent `Date_Received` across the donor's donations.
- **`ImpactSummary`** — a one-line generated message using the item count.
- **`NextPendingItem`** — the donor's soonest-due pending pledge item.
- **`PledgeCompletionRate`** — fulfilled vs. total pledge count as a percentage.
- **`NextPickupDateTime`** — the donor's soonest pending pickup date.
- **`PickupLocation`** — see "left honest" note below.

I also fixed the `MessageBox.Show("Failed to load...")` calls in
`InitializeTotalDonationsCount`, `InitializeActivePledgesCount`, and
`InitializeScheduledPickupsCount`. These fired on *any* zero-row result —
including the completely normal case of a brand-new donor with nothing yet
— same false-error pattern I fixed elsewhere on the donation side. They're
removed; the counters just correctly show 0.

All three new "card detail" methods are wired into `RefreshCounters()`, so
they update on login and whenever a pledge is submitted (via the existing
`DonationDataChanged` event from the previous pass).

## Left honest, not fabricated

- **`PickupLocation`** always shows "Not tracked yet." I checked the
  `Pickup Schedule` table schema — it only has `Pickup_ID`, `Donation_ID`,
  `Pickup_Date`, `Pickup_Status`. There's no location or center column at
  all, so there's nothing real to query. Showing a location here would mean
  inventing data.
- **`NextDropOffDateTime`, `DestinationHub`, `DeliveryStatus`** (the
  "Scheduled Deliveries" card) stay at placeholder values ("Not yet
  scheduled" / "—"). This is the same gap flagged on Day 1:
  `Delivery Schedule` has no FK path back to a donor's `Donation` — it only
  connects to `Distribution`, which is the outgoing (Beneficiary/Center)
  side. Nothing changed here; still pending your schema decision.

## To apply

Just replace `donationdashboard_mainlayout_ViewModel.cs` with the updated
version in this folder. No SQL changes needed for this particular fix — the
new queries use `GetTableDataWithCustomizedQuery`, which doesn't require
new stored procedures or functions, just inline SQL built with manual
single-quote escaping (no parameterization available on that method, so I
escaped `'` → `''` by hand to keep it injection-safe).

After this, log in as donor1 again — `TotalItemCount`, `LastDonationDate`,
and `ImpactSummary` should now show real values reflecting the 2 existing
donations.
