## What was actually broken, and what I fixed

### 1. Dashboard counters never refreshed (the original Day 2 item)
`MyDonations`, `MyPledges`, `MakeAPledge`, and `Pickup&Delivery` are all
created fresh by the layout's navigation commands (`new View.xxx_view()`),
so they have no reference back to the dashboard's counters. Submitting a
pledge never updated "Active Pledges" until the user logged out and back in.

**Fix:** added a static event, `donationdashboard_mainlayout_ViewModel.DonationDataChanged`.
`MakeAPledge_ViewModel.SubmitPledge()` now calls
`donationdashboard_mainlayout_ViewModel.NotifyDonationDataChanged()` right
after a successful pledge insert. The layout VM subscribes to this event in
its constructor and unsubscribes on logout (to avoid stacking handlers if
the same user logs in and out multiple times in one app session).

### 2. `MakeAPledge` had a wrong `using` directive
The file had `using System.Windows.Forms;` instead of `using System.Windows;`.
This meant every `MessageBox.Show(...)` in that file was silently resolving
to the WinForms MessageBox, not WPF's — in a WPF app, that's a mismatched UI
framework reference. Fixed.

### 3. Debug leftovers removed
- `MessageBox.Show(SelectedDisasterEvent.Name)` in the property setter —
  removed (this would have shown a debug popup every single time a user
  picked a disaster event from the dropdown).
- Dead commented-out `//MessageBox.Show(...)` lines in `MyDonations` and
  `MyPledges` filter/search setters — removed for cleanliness.

### 4. `PledgeItems[0]` with no bounds check
`SubmitPledge()` accessed `PledgeItems[0]` directly. If a user clicked
Submit without first clicking "Add Item," this throws an unhandled
`IndexOutOfRangeException` and crashes that part of the UI. Now checks
`PledgeItems.Count == 0` first and shows a normal validation message instead.

### 5. `SubmitPledge` was `async void`, not `async Task`
Fire-and-forget — any exception inside it would crash the app instead of
being catchable, and there was no way to `await` it from the command. Changed
to `async Task` and the `RelayCommand` now does
`new RelayCommand(async () => await SubmitPledge())`. Same fix applied to
`donation_signup_ViewModel.Signup()` and `rescuer_signup_ViewModel.Signup()`
back on Day 1.

### 6. False "error" popups on empty results
`MyDonations`, `MyPledges`, and `Pickup&Delivery` all showed a MessageBox
like "No available donations" whenever the query returned zero rows — but
zero rows is the **normal** state for a brand-new donor who hasn't pledged
or donated anything yet, not a database error. These MessageBoxes are
removed; an empty `ObservableCollection` now just renders as an empty list
in the UI (assuming the XAML has reasonable empty-state styling — if it
currently shows nothing at all when the list is empty, that's a small XAML
follow-up, not a backend issue).

### 7. `MyDonations` never loaded on first view
Unlike `MyPledges`, the `MyDonations` constructor never called
`InitializeMyDonations()`, and the old code showed a blocking "Please select
a donation filter" popup the moment the user did interact with the filter
dropdown, since nothing defaulted it. Now both `MyDonations` and `MyPledges`
default to `"All Donations"` / `"All Pledges"` respectively when no filter is
selected yet, and load automatically when the view is constructed.

### 8. Pledge form doesn't reset after submission
After a successful `SubmitPledge()`, the form previously left the old
pledge item, event selection, and date sitting in the UI — easy to
accidentally double-submit the same pledge. Now clears `PledgeItems`,
`SelectedDisasterEvent`, and resets the date after a successful submit.

### 9. Expected delivery date validation added
`SubmitPledge()` now rejects a delivery date in the past, with a clear
message, instead of letting it silently insert a backdated pledge.

## Something I found but did NOT silently fix — needs your decision

The `MyDonations` filter dropdown (XAML) offers **"Pending," "Received,"
"Completed"** as `Pickup_Status` filter options. I checked the actual seed
data in your `.bacpac`, and the only `Pickup_Status` values that exist are
**`Pending`** and **`Completed`** — there's no `Received` anywhere in the
data. That means the "Received" filter, as it stands, will always return
zero results, every time, for every user — not a code bug, but a real
mismatch between the UI options and what the database will ever produce
for that column. I didn't touch the XAML or invent a value, since I can't
tell whether "Received" is meant to be a future planned status, a renamed
existing status, or a leftover that should be removed. Worth deciding before
you ship this screen.
