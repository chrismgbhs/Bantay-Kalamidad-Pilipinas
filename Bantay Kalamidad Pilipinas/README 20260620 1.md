# Day 1 — Thursday, June 18 — Backend Summary

## What changed this turn

You uploaded the actual `.bacpac` (database export), which let me check the
real schema instead of guessing from the C# code. That uncovered the actual
root cause behind most of the "No available..." / "Failed to load..."
messages in the app:

**All 15 table-valued functions the ViewModels call already exist in the
database as objects — but every single one has an empty body.** No `SELECT`,
no `RETURN`, nothing. `GetTableData()` isn't failing on bad SQL; it's
successfully calling a function that returns zero rows by design, because the
function was never written.

This is good news — it means the C# side is already wired correctly to call
the right function names with the right parameters. The fix is 100% on the
SQL side.

## Things worth knowing

- **`usp_AddDonor` / `usp_AddVolunteer` already check for duplicate
  usernames** via `RAISERROR`, and there's *also* an `INSTEAD OF INSERT`
  trigger (`trg_PreventDuplicateUser`) on `Users` doing the same check. Two
  layers catching the same thing isn't a bug, just redundant — the procedure's
  check fires first since it runs before the `INSERT` that would trigger the
  trigger. Not worth touching today; flagged for later cleanup if you want it.
- **`GetDeliveryStatusByUsername` is left returning an empty result on
  purpose.** I traced the FK chain in the schema and there's currently no
  path from a donor's `Donation` row to a `Delivery Schedule` row —
  `Delivery Schedule` hangs off `Distribution`, which is the *outgoing* side
  (Beneficiary → Event → Center), not the donor's incoming donation. This
  needs an actual schema/business decision (e.g. a `Donation_ID` column on
  `Delivery Schedule`, or a join through `Pickup Schedule`) before it can be
  wired up — I didn't want to guess and write something that looks like it
  works but returns wrong data. Flagged for you to decide before Day 2.
- **`GetMyDonations` and `GetMyPledgesByUsername` return one row per item**,
  not one row per donation/pledge. Since `Item_Name` lives in a child table
  (`Donated Items` / `Pledge Item`) and a single donation or pledge can have
  multiple items, a donation with 3 items will show as 3 rows. That matches
  what the ViewModel's `ObservableCollection<Donation>` / `MyPledge` already
  expect (one item name per row), so no C# changes were needed — just flagging
  it so the behavior doesn't look like a bug when you test it.
