# Google Sign-In — Setup &amp; Implementation

"Sign in with Google" is now wired up on all four screens (Donor login,
Donor signup, Volunteer login, Volunteer signup). Before it'll actually
work, there are two things only you can do — neither is something I can do
on your behalf, since both require your own Google account and your actual
Visual Studio project.

---

## Part 1 — Google Cloud Console setup (you do this once)

1. Go to **[console.cloud.google.com](https://console.cloud.google.com)**
   and sign in.
2. Click the project dropdown at the top → **New Project**. Name it
   something like "Bantay Kalamidad Pilipinas" → **Create**.
3. Once the project is selected, go to **APIs & Services → OAuth consent
   screen**.
   - User Type: **External** (unless you have a Google Workspace
     organization, in which case Internal is fine and skips verification).
   - Fill in the App name ("Bantay Kalamidad Pilipinas"), your email as
     support contact, and your email again under developer contact info.
   - Scopes: add `openid`, `.../auth/userinfo.email`,
     `.../auth/userinfo.profile` (these are the three this app requests).
   - Test users: while the app is unverified, add the Google account(s) you
     and your team will test with here, or sign-in will be blocked for
     anyone not listed.
4. Go to **APIs & Services → Credentials → Create Credentials → OAuth
   client ID**.
   - Application type: **Desktop app** (this is important — Web application
     won't work correctly for a WPF desktop app's redirect flow).
   - Name it anything, e.g. "BKP Desktop Client".
   - Click **Create**. A dialog shows your **Client ID** and **Client
     Secret** — copy both.

That's it on the Google side. No redirect URI configuration is needed for
the Desktop app client type — `Google.Apis.Auth` picks a random local port
automatically at runtime.

## Part 2 — Add the NuGet package (you do this in Visual Studio)

This project uses the older `packages.config`-style setup (not the newer
SDK-style `PackageReference`), so:

1. In Visual Studio, right-click the project → **Manage NuGet Packages**.
2. Search for **`Google.Apis.Auth`** → Install (this pulls in
   `Google.Apis.Oauth2.v2` and `Google.Apis.Core` as dependencies
   automatically).
3. If prompted to review changes, accept.

(This must be done in Visual Studio's NuGet UI or Package Manager Console —
I can't run NuGet against your actual project from here.)

## How it works

- Clicking "Sign in with Google" opens your **default system browser** (not
  an embedded browser — Google explicitly disallows those for security
  reasons) to Google's real sign-in page.
- After the user signs in and consents, the browser redirects to a random
  `localhost` port your app is briefly listening on. This is Google's
  officially recommended flow for desktop apps (RFC 8252).
- The app reads the user's verified email and name from Google, then:
  - **If a Users row already exists** with that email → logs them straight
    into the matching dashboard (Donor or Volunteer, whichever button they
    clicked).
  - **If no account exists yet** → automatically creates one via the
    existing `usp_AddDonor` / `usp_AddVolunteer` procedures, using their
    Google name and a long random placeholder password (see note below),
    shows a one-time "Welcome, &lt;name&gt;!" message, then opens the
    dashboard.
  - **If the email is already registered under the other role** (e.g. they
    click "Sign in with Google" on the Donor screen but their account is a
    Volunteer) → blocked with a clear "wrong portal" message, same
    behavior as the existing password login.

## About the placeholder password

Google accounts don't have a BKP password, but `Users.Password` is a
required (`NOT NULL`) column. A 256-bit cryptographically random value is
generated, hashed the same way a real password would be, and stored. It's
never shown anywhere and the password login form can never produce a
matching hash for it — so a Google-created account can genuinely only ever
be accessed by signing in with that same Google account again.

## What I could not verify myself

I don't have a Windows machine, Visual Studio, a real Google account, or
your actual database to test this against — everything here is written
against Google's official documentation and the actual `Google.Apis.Auth`
library API, but **you'll be the first one to actually run this flow
end-to-end**. A few things worth testing deliberately once it's wired up:

1. First-time sign-in on the Donor screen → confirm the welcome message
   appears and a real `Users`/`Donor` row gets created.
2. Sign out, sign back in with the same Google account → confirm it logs
   straight in with no duplicate account or welcome message the second
   time.
3. Try signing in with the same Google account on the *Volunteer* screen →
   confirm you get the "already registered as a Donor" message rather than
   a duplicate or broken account.
4. Click "Sign in with Google" and then close the browser tab without
   completing sign-in → confirm the app shows the "cancelled" message
   instead of crashing.

If any of these don't behave as described, it's most likely something in
the Google Cloud Console consent screen configuration (e.g. forgetting to
add a test user while the app is unverified) rather than the C# — that's
the part most prone to a small setup mistake on the Google side.
